# AGENTS.md

This file provides guidance to AI coding agents (Claude Code, and others) when working with code in this repository.

## What this project is

Open-source AI memory platform: indexes knowledge sources (Obsidian, Git, PDFs), generates semantic embeddings via Qdrant, and injects the most relevant context into LLM queries.

## Architecture

### Request flow

```
Next.js frontend â†’ Go API (Chi router) â†’ Retrieval engine â†’ Qdrant (vector search)
                                       â†’ Context compression â†’ LLM provider â†’ SSE stream
                                       â†’ PostgreSQL (metadata/state)
```

### Backend layout (`api/`)

The API follows a standard Go `internal/` layout. Each feature domain lives under `api/internal/<domain>/` and exposes three files: `handler.go` (HTTP layer), `service.go` (business logic interface + implementation), `routes.go` (Chi subrouter). The pattern for wiring a new domain:

1. Create `internal/<domain>/{handler,service,routes}.go`
2. In `router/router.go`, instantiate the service and handler, then mount routes with `r.Mount("/<path>", domain.Routes(handler))`

### Configuration (`api/internal/config/`)

Config is loaded once at startup via `sync.Once` and validated with `go-playground/validator`. Environment variables use a `<PREFIX>_<FIELD>` naming convention driven by `kelseyhightower/envconfig`:

| Struct | Env prefix |
|--------|-----------|
| `Server` | `SERVER_` |
| `Postgres` | `POSTGRES_` |
| `Qdrant` | `QDRANT_` |
| `Ollama` | `OLLAMA_` |
| `MinIO` | `MINIO_` |

In non-production environments, `.env` is loaded first, then `.env.<SERVER_ENV>` overrides it. In production, only system env vars are used.

### Logging

All logging goes through `log/slog`. The global logger is initialised once in `main.go` via `logger.Init(level)` and emits structured JSON to stdout.

### Swagger

API docs are auto-generated by `swaggo/swag`. Annotations live on handler methods. The docs are served at `/swagger/`. To regenerate after changing annotations: `make swag` from the repo root. CI (`ci-api.yml`) fails the build if `api/docs` is out of sync with the annotations.

### Providers

Both embeddings and LLM inference are pluggable by design â€” no provider is hardcoded. Default: `nomic-embed-text` via Ollama for embeddings. LLM providers: OpenAI, Claude, Gemini, Ollama, Qwen, DeepSeek.

## Conventions

**Commits:** Conventional Commits â€” `<type>(<scope>): <summary>` (e.g. `feat(chunking): add markdown section splitter`)

**Branches:** `feat/`, `fix/`, `docs/`, `refactor/`, `chore/` prefixes off `main`

**PRs:** squash merge; one concern per PR; fill the PR template and link the related issue

**ADRs:** significant technical decisions go in `docs/adr/` using the template at `docs/adr/ADR-XXX-template.md`

**Specs:** technical specs live in `docs/specs/` using the template at `docs/specs/SPEC-XXX-template.md`, drafted via the `/spec` command â€” ADRs record decisions, specs describe components (implemented or planned)

## Constraints

- Do NOT add new dependencies without explicit instruction â€” evaluate stdlib or existing deps first
- Do NOT modify `docs/adr/` without being asked â€” ADRs are written by humans
- Do NOT write to `docs/specs/` without confirmation â€” specs are drafted via `/spec` and approved by a human

- Do NOT run `make swag` / `swag init` automatically â€” only when explicitly asked
- Do NOT commit directly to `main`

## Code patterns

### Error handling
Wrap errors with context using `fmt.Errorf("operation: %w", err)`. Services return domain errors; handlers translate them to HTTP status codes.

### Tests
Table-driven tests with `t.Run`. Config tests must call `resetGlobals()` (defined in `config_test.go`, same package) between cases to reset the singleton. No mocks for external services â€” use real Qdrant/Postgres via Docker in integration tests.

## Project status

### Implemented
- Config loading (`internal/config/`)
- Health endpoint (`internal/health/`) â€” handler, service, routes mounted at `/health`
- CI pipelines (lint, test, build)
- PostgreSQL connection pool (`internal/storage/`) â€” `Pool` interface backed by `pgxpool`, migrations under `storage/postgres/migrations/`
- Qdrant client (`internal/vectorstorage/`) â€” `Client` interface with `Upsert`, `Query`, `Delete`, `Count`; `ensureCollection` runs on startup
- Chunking engine (`internal/chunking/`) â€” `Splitter` interface + `ChunkService` with `ChunkSource`, `ListChunks`, `DeleteChunks`; concrete splitters under `chunking/markdown/` and `chunking/text/`
- Source reader (`internal/sourcereader/`) â€” `Reader` interface + `FileReader` that walks a local directory and maps files to `ChunkRequest` values
- Object storage (`internal/objectstorage/`) â€” `Client` interface backed by MinIO (S3-compatible); `ensureBucket` runs on startup
- Sources endpoints (`internal/sources/`) â€” handler, service, routes mounted at `/sources`
  - `POST /sources` â€” multipart upload â†’ MinIO â†’ 202 Accepted; indexing runs in background
  - `POST /sources/{id}/ingest` â€” re-download from MinIO and re-index in background
  - `GET /sources/{id}/status` â€” SSE stream of indexing progress (heartbeat every 30s, timeout 15min)
  - `GET /sources?workspace_id=` â€” list sources for a workspace
  - `GET /sources/{id}/chunks` â€” list indexed chunks for a source
- Embedding generation (`internal/embedding/`) â€” `Embedder` interface + Ollama-backed implementation (`internal/embedding/ollama/`) for `nomic-embed-text`; `embedding.NewEmbedder` factory mirrors `vectorstorage.NewClient`; shared types in `embedding/types/` break the import cycle
- Ingestion pipeline (`internal/sources/`) â€” after chunking, `runPipeline` calls `EmbedBatch`, validates vectors, upserts to Qdrant (point ID = chunk UUID; payload: `chunk_id`, `workspace_id`, `source_id` only), and batch-updates `chunks.embedding_model` in Postgres
