# AGENTS.md

This file provides guidance to AI coding agents (Claude Code, and others) when working with code in this repository.

## What this project is

Open-source AI memory platform: indexes knowledge sources (Obsidian, Git, PDFs), generates semantic embeddings via Qdrant, and injects the most relevant context into LLM queries.

## Commands

### Infrastructure (required before running the API or frontend locally)

```bash
docker compose up -d qdrant postgres ollama minio
ollama pull nomic-embed-text
ollama pull llama3
```

### Backend (Go API)

```bash
cd api
go run ./cmd/server          # start the API at :8080
go test ./...                # run all tests
go test ./... -race          # run tests with race detector (matches CI)
go test ./internal/config/... # run a specific package
go build ./cmd/server        # build
golangci-lint run            # lint
```

### Frontend (web)

Thin Next.js (App Router, TypeScript) client. Requires the API running (proxied via `/api/*`). Sign in with the bundled Keycloak dev user `dev` / `dev`.

```bash
cd web
npm install
npm run dev                  # http://localhost:3000
npm run lint                 # next lint
npm run type-check           # tsc --noEmit
npm run test                 # vitest run
npm run test:coverage        # vitest run --coverage (matches CI's test-coverage-web.yml)
npm run build                # production build
```

`next.config.mjs` rewrites `/api/*` to the API (`API_BASE_URL`, default `http://localhost:8080`) so browser and API share an origin ŌĆö the session cookie is first-party and no CORS is needed.

### Environment setup

Two env templates: the repo root serves Docker Compose interpolation; `api/` serves the application.

```bash
cp .env.example .env          # Docker Compose (ports, service credentials)
cp api/.env.example api/.env  # Go API config (read from api/ at startup)
```

## Architecture

### Request flow

```
Next.js frontend ŌåÆ Go API (Chi router) ŌåÆ Retrieval engine ŌåÆ Qdrant (vector search)
                                       ŌåÆ Context compression ŌåÆ LLM provider ŌåÆ SSE stream
                                       ŌåÆ PostgreSQL (metadata/state)
```

### Backend layout (`api/`)

The API follows a standard Go `internal/` layout. Each feature domain lives under `api/internal/<domain>/` and exposes three files: `handler.go` (HTTP layer), `service.go` (business logic interface + implementation), `routes.go` (Chi subrouter). The pattern for wiring a new domain:

1. Create `internal/<domain>/{handler,service,routes}.go`
2. In `router/router.go`, instantiate the service and handler, then mount routes with `r.Mount("/<path>", domain.Routes(handler))`

### Configuration (`api/internal/config/`)

Config is loaded once at startup via `sync.Once` and validated with `go-playground/validator`. Environment variables use a `<PREFIX>_<FIELD>` naming convention driven by `kelseyhightower/envconfig`:

| Struct | Env prefix |
|--------|-----------|
| `Server` | `SERVER_` |
| `Postgres` | `POSTGRES_` |
| `Qdrant` | `QDRANT_` |
| `Ollama` | `OLLAMA_` |
| `MinIO` | `MINIO_` |
| `Auth` | `AUTH_` |
| `Reranker` | `RERANKER_` |

In non-production environments, `.env` is loaded first, then `.env.<SERVER_ENV>` overrides it. In production, only system env vars are used.

### Logging

All logging goes through `log/slog`. The global logger is initialised once in `main.go` via `logger.Init(level)` and emits structured JSON to stdout.

### Swagger

API docs are auto-generated by `swaggo/swag`. Annotations live on handler methods. The docs are served at `/swagger/`. To regenerate after changing annotations: `make swag` from the repo root. CI (`ci-api.yml`) fails the build if `api/docs` is out of sync with the annotations.

### Providers

Embeddings, LLM inference, and reranking are all pluggable by design ŌĆö no provider is hardcoded. Default: `nomic-embed-text` via Ollama for embeddings. LLM providers: OpenAI, Claude, Gemini, Ollama, Qwen, DeepSeek. Reranking: Hugging Face Text Embeddings Inference (`internal/reranking`), required at startup like every other provider.

## Conventions

**Commits:** Conventional Commits ŌĆö `<type>(<scope>): <summary>` (e.g. `feat(chunking): add markdown section splitter`)

**Branches:** `feat/`, `fix/`, `docs/`, `refactor/`, `chore/` prefixes off `main`

**PRs:** squash merge; one concern per PR; fill the PR template and link the related issue

**ADRs:** significant technical decisions go in `docs/adr/` using the template at `docs/adr/ADR-XXX-template.md`

**Specs:** technical specs live in `docs/specs/` using the template at `docs/specs/SPEC-XXX-template.md`, drafted via the `/spec` command ŌĆö ADRs record decisions, specs describe components (implemented or planned)

## Claude Code skills

`.claude/skills/` holds project-committed Skills that Claude Code triggers automatically from context, unlike the explicit `/command`s in `.claude/commands/` (`/architecture`, `/commit`, `/issue`, `/pr`, `/review`, `/spec`). Keep each in sync with the code, docs, or CI config it describes as those change.

**Workflow:**
- **`run`** ŌĆö the local launch sequence (Docker infra ŌåÆ migrations ŌåÆ API ŌåÆ frontend ŌåÆ sign-in), plus the CLI's current auth limitation
- **`verify`** ŌĆö mirrors the actual CI gates (`ci-api.yml`, `ci-web.yml`, the swagger-drift check) and how to exercise the affected flow live before calling a change done
- **`code-review`** ŌĆö auto-triggered version of `/review`'s convention check against `.claude/anti-patterns.md` and `.claude/glossary.md`

**Backend patterns:**
- **`go-feature-domain`** ŌĆö the three-file domain layout, its justified exceptions, and testing conventions
- **`go-provider-interface`** ŌĆö the pluggable-backend pattern (`embedding`, `llm`, `vectorstorage`, `objectstorage`, `storage`), its fail-fast startup convention, and two known wiring inconsistencies (`chunking`, `sourcereader`)

**Backend domains with non-obvious behavior:**
- **`auth`** ŌĆö the three login surfaces and the custom (non-JWT) session cookie scheme
- **`sources-ingestion`** ŌĆö in-process indexing, stuck-index recovery, the re-ingest CAS guard, and path normalization
- **`retrieval`** ŌĆö defense-in-depth workspace filtering, rank-based re-sorting after hydration, `topK` clamping
- **`workspaces`** ŌĆö why `EnsureMember` is a manual guard, not middleware, and that roles aren't enforced yet

**Infrastructure / technology:**
- **`postgres-migrations`** ŌĆö migrations run through a separate `cmd/migrate` binary, never automatically by the server
- **`qdrant`** ŌĆö the intentionally leaky, protobuf-typed vector store interface (ADR-008)
- **`ollama`** ŌĆö the fail-fast model check and why the LLM client has no HTTP timeout
- **`swagger`** ŌĆö the annotation ŌåÆ `make swag` ŌåÆ CI drift-check flow

**Frontend:**
- **`web-conventions`** ŌĆö the `lib/api/` ŌåÆ `hooks/` (TanStack Query) ŌåÆ `components/` layering, and the SSE exception to it
- **`shadcn-ui`** ŌĆö the official shadcn/ui CLI (`npx shadcn@latest add ...`) and this project's actual `web/components.json` config; no third-party registries or paid services
- **`tanstack-query`** ŌĆö `@tanstack/react-query` v5 API reference (queries, mutations, invalidation, SSR/hydration) backing the `web-conventions` hook layer

## Constraints

- Do NOT add new dependencies without explicit instruction ŌĆö evaluate stdlib or existing deps first
- Do NOT modify `docs/adr/` without being asked ŌĆö ADRs are written by humans
- Do NOT write to `docs/specs/` without confirmation ŌĆö specs are drafted via `/spec` and approved by a human
- Do NOT change `api/.env.example` without updating this file's config table (the root `.env.example` holds only Docker Compose interpolation vars)
- Do NOT run `make swag` / `swag init` automatically ŌĆö only when explicitly asked
- Do NOT commit directly to `main`

## Code patterns

### Error handling
Wrap errors with context using `fmt.Errorf("operation: %w", err)`. Services return domain errors; handlers translate them to HTTP status codes.

### Tests
Table-driven tests with `t.Run`. Config tests must call `resetGlobals()` (defined in `config_test.go`, same package) between cases to reset the singleton. No mocks for external services ŌĆö use real Qdrant/Postgres via Docker in integration tests.

## Project status

### Implemented
- Config loading (`internal/config/`)
- Health endpoint (`internal/health/`) ŌĆö handler, service, routes mounted at `/health`
- CI pipelines (lint, test, build)
- PostgreSQL connection pool (`internal/storage/`) ŌĆö `Pool` interface backed by `pgxpool`, migrations under `storage/postgres/migrations/`
- Qdrant client (`internal/vectorstorage/`) ŌĆö `Client` interface with `Upsert`, `Query`, `Delete`, `Count`; `ensureCollection` runs on startup
- Chunking engine (`internal/chunking/`) ŌĆö `Splitter` interface + `ChunkService` with `ChunkSource`, `ListChunks`, `DeleteChunks`; concrete splitters under `chunking/markdown/` and `chunking/text/`
- Source reader (`internal/sourcereader/`) ŌĆö `Reader` interface + `FileReader` that walks a local directory and maps files to `ChunkRequest` values
- Object storage (`internal/objectstorage/`) ŌĆö `Client` interface backed by MinIO (S3-compatible); `ensureBucket` runs on startup
- Sources endpoints (`internal/sources/`) ŌĆö handler, service, routes mounted at `/sources`
  - `POST /sources` ŌĆö multipart upload ŌåÆ MinIO ŌåÆ 202 Accepted; indexing runs in background
  - `POST /sources/{id}/ingest` ŌĆö re-download from MinIO and re-index in background
  - `GET /sources/{id}/status` ŌĆö SSE stream of indexing progress (heartbeat every 30s, timeout 15min)
  - `GET /sources?workspace_id=` ŌĆö list sources for a workspace
  - `GET /sources/{id}/chunks` ŌĆö list indexed chunks for a source
- Embedding generation (`internal/embedding/`) ŌĆö `Embedder` interface + Ollama-backed implementation (`internal/embedding/ollama/`) for `nomic-embed-text`; `embedding.NewEmbedder` factory mirrors `vectorstorage.NewClient`; shared types in `embedding/types/` break the import cycle
- Reranking (`internal/reranking/`) ŌĆö `Reranker` interface + Hugging Face Text Embeddings Inference-backed implementation (`internal/reranking/tei/`) for cross-encoder scoring; `reranking.NewReranker` factory mirrors `embedding.NewEmbedder`; shared types in `reranking/types/` break the import cycle
- Ingestion pipeline (`internal/sources/`) ŌĆö after chunking, `captionStructuredChunks` (`internal/sources/caption.go`) appends an LLM-generated caption to any chunk containing a fenced code block or Markdown table (updating both the persisted row and the in-memory copy), then `runPipeline` calls `EmbedBatch`, validates vectors, upserts to Qdrant (point ID = chunk UUID; payload: `chunk_id`, `workspace_id`, `source_id` only), and batch-updates `chunks.embedding_model` in Postgres
- Retrieval engine (`internal/retrieval/`) ŌĆö `Retriever` interface + `RetrievalService`; `POST /query` embeds the question, runs a hybrid search (vector search against Qdrant fused with Postgres full-text search over `chunks.content_tsv` via Reciprocal Rank Fusion), reranks the fused candidates with the cross-encoder, and hydrates the top-k chunks from Postgres (see [SPEC-006](docs/specs/SPEC-006-retrieval-engine.md))
- CLI (`cmd/cli/`) ŌĆö minimal `ingest`/`query` commands that talk to the API as a plain HTTP client (no `internal/*` imports); `make build-cli` builds `dist/nv` (falls back to `dist/neuralvault` if `nv` is already on `PATH`), `make run-cli ARGS=...` runs it via `go run`
- LLM provider (`internal/llm/`) ŌĆö `Provider` interface + Ollama-backed implementation (`internal/llm/ollama/`) for chat completion (`Complete` and `Stream`); `llm.NewProvider` factory mirrors `embedding.NewEmbedder`; shared types in `llm/types/` break the import cycle (see [SPEC-007](docs/specs/SPEC-007-llm-provider-layer.md))
- Auth + identity (`internal/auth/`) ŌĆö OIDC authorization-code login (Keycloak in dev, any OIDC provider by `AUTH_` config), JIT user provisioning, HMAC-signed `nv_session` cookie, and `RequireUser` middleware; `/auth/*` routes, with `/sources`/`/query`/`/workspaces` protected (see [SPEC-011](docs/specs/SPEC-011-auth-workspaces-tenant-isolation.md))
- Workspaces + tenant isolation (`internal/workspaces/`) ŌĆö `POST`/`GET /workspaces` plus an `EnsureMember` guard enforcing membership (403) on every workspace-scoped route
- Streaming answers ŌĆö `POST /query/stream` runs retrieval, builds a RAG prompt, and streams a grounded LLM answer over SSE (`sources`/`token`/`done` events); non-streaming `POST /query` remains for the CLI
- Frontend (`web/`) ŌĆö thin Next.js (App Router, TypeScript) chat client: OIDC sign-in gate, workspace switcher, streaming chat, and file upload with live indexing status; proxies `/api/*` to the API (see `web/README.md`)
